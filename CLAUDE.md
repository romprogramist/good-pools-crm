# CRM «Хорошие Бассейны» — для Claude

Этот файл загружается автоматически при работе в этой папке.

## Контекст

Web-CRM для компании «Хорошие Бассейны», обслуживающей частные бассейны в Сочи.
Это **отдельный** проект от `~/good_pools/` (тот — лендинг компании).

## Где что лежит

- `spec.md` — полная спецификация и решения, утверждённые клиентом 2026-05-08.
- `plan.md` — план реализации по 17 этапам с галочками. **При старте новой сессии — читай этот файл первым делом**, чтобы понять где остановились.
- `src/` — код Next.js приложения (после этапа 1).
- `prisma/` — схема БД и миграции (после этапа 2).
- `worker/` — отдельный процесс с cron-задачами (после этапа 15).

## Как продолжать работу

1. Прочитай `plan.md` — найди первый незавершённый этап.
2. Скажи юзеру что-то вроде: «Сейчас в работе этап N — <название>. Продолжаем?»
3. Реализуй подзадачи этапа.
4. После каждой подзадачи — отмечай галочку `[x]` в `plan.md`.
5. Когда этап готов — сформулируй чёткий чекпойнт юзеру: «Открой X, проверь Y». Дождись подтверждения.
6. Только после подтверждения — переходи к следующему этапу.

## Важные технические решения

- Стэк: Next.js 16 + TS + Prisma + Postgres + Tailwind + shadcn/ui + Auth.js
- Postgres локальный на 5432 (не запускать в Docker на Mac, не использовать другие БД проектов как fallback)
- Push: Web Push + VAPID + Serwist (PWA)
- PDF: @react-pdf/renderer (НЕ Puppeteer)
- Карты: Яндекс.Карты JS API v3
- Чат: поллинг 5 сек (НЕ WebSocket)
- Деплой: Docker Compose на сервере клиента (НЕ Vercel — multi-tenancy не нужна)

## Что НЕ делать

- Не предлагать переключиться на Supabase / Vercel / другие managed сервисы — у клиента свой сервер.
- Не делать SMS-уведомления — только Web Push.
- Не делать импорт данных — стартуем с пустой базы.
- Не закладывать multi-tenancy — продажа другим компаниям = отдельная установка.
- Не сохранять историю изменений индивидуальной цены сервиса — клиент явно отказался.
- Не использовать Puppeteer для PDF.
